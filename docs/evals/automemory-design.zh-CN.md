# AutoMemory 评估设计

[English](automemory-design.md) | [简体中文](automemory-design.zh-CN.md)

> **协议快照：** 2026-07-07 评估汇总
>
> **证据类型：** 真实模型、小样本实验
>
> **公开结果：** [AutoMemory 评估报告](automemory-report.zh-CN.md)

## 术语表

| 术语 | 定义 |
|---|---|
| AutoMemory | 一条可选启用的跨会话记忆路径：它会在一次会话后提取持久信息，并可在后续会话中将其呈现出来。 |
| S1 | 教学会话；处理组在此接收可能值得保留的信息。 |
| S2 | 全新的探测会话；智能体必须在任务不再重复该信息的情况下应用它。 |
| 处理组（A） | 启用 AutoMemory。只有 S1 产生的记忆制品会被转移到全新的 S2 环境。 |
| 对照组（B） | 禁用 AutoMemory。S2 既不接收 S1 记忆，也不接收 S1 会话状态。 |
| 写入门禁（Write gate） | 针对具体用例的检查，用于确认预期信息在 S2 评分前确实已经存储。 |
| 写入成功率（Write success） | 合格处理组运行中，S1 记忆通过写入门禁的比例。 |
| 写入后的召回率（Recall given write） | 在通过写入门禁的处理组运行中，S2 结果通过其评分器的比例。 |
| 处理组端到端通过率（End-to-end treatment pass） | 合格处理组运行中，既正确写入记忆、又正确应用记忆的比例。 |
| 差值（`delta`） | 对增量型用例，计算方式为 `recall_given_write - control_pass_rate`。精确性单独报告，因为它的期望差值约为零。 |
| 精确性用例（Precision case） | 一种带作用域的记忆用例；正确行为是不把 PaymentService 约定应用到无关的 OrderRepo 任务。 |
| 验证性对照（Foil） | 在认可“忽略条件”成功抑制记忆之前，用来证明该记忆确实能够影响行为的条件。 |
| `PASS` | 有效证据满足声明的用例预期。 |
| `FAIL` | 有效证据未满足声明的用例预期。 |
| `INVALID` | 必需的实验前提未满足，因此该样本或用例无法区分目标行为。 |
| `INCONCLUSIVE` | 没有可用样本，或现有证据不足以作出判断。 |
| `ERROR` | 运行时、模型调用或评分器失败；这不是行为结论。 |
| `OPEN` | 后续决策状态，而非已评分样本的状态。该用例仍未解决，且不计入正向结论。 |

## 摘要

该协议将记忆获取与后续使用分开。增量型用例比较启用记忆与禁用记忆两种条件；精确性用例检查带作用域的记忆是否会泄漏到无关任务；行为用例测试过期记忆纠正、显式忽略与临时信息拒绝。无效设置和有效样本数为零的条件会被排除，既不会被计作模型失败，也不会被静默重采样。

## 本评估回答的问题

1. 与无记忆对照相比，S1 写入的信息是否会改变全新 S2 会话中的行为？
2. 一次失败是因为记忆没有写入，还是因为已写入的记忆没有得到应用？
3. 一个无关的已存储约定是否会降低智能体在另一个模块上的表现？
4. 智能体能否优先采用当前仓库事实而非过期记忆、服从显式忽略指令，并拒绝普通临时上下文？

本评估不估计广泛任务总体上的表现，不比较不同产品，不证明所有记忆都能被精确存储，也不确立生产级可靠性。

## 实验结构

### Track 1A：增量 A/B

处理组与对照组接收相同的 S1 和 S2 任务。两者的预期差异是 S2 中是否可用跨会话记忆。

| 用例 | 持久信息 | 全新会话探测 | 期望信号 |
|---|---|---|---|
| `H_ref` | 历史摄取上下文位于一个指定的 Linear 项目中。 | 询问应在哪里查找该上下文。 | 处理组给出两个私有标识符；对照组通常无法给出。 |
| `H_proj` | 在声明日期之后冻结非关键合并。 | 提议一次更晚的非关键合并。 | 处理组识别出冲突；对照组不知道该政策。 |
| `H_fb2` | 重构应通过一个合并后的 PR 交付。 | 请求一个自然包含三个拆分点的计划。 | 处理组保持为一个 PR；对照组倾向于拆分工作。 |
| `H_usr` | 用户熟悉 Go，但刚开始学习 React。 | 解释一个 React 组件的状态流。 | 处理组使用具体的 Go 类比；泛泛的初学者友好说明并不足够。 |
| `H_fb1` | 数据库测试不应模拟数据库。 | 请求编写一个 OrderRepo 测试。 | 该用例保留为一条 `INVALID` 设计经验，因为对照组自然选择了相同行为。 |

增量差值仅针对有效且具有区分力的用例计算。`H_fb1` 从不计入汇总范围。

### Track 1B：精确性 A/B

`H_prec` 首先存储一条仅适用于 PaymentService 测试的约定，随后要求两种条件测试 OrderRepo，而 OrderRepo 没有支付网关依赖。只有当诱饵约定确实已经存储时，处理组才有效。期望结果为：

```text
treatment_pass_rate approximately equals control_pass_rate
delta approximately equals 0
```

负差值将表明记忆误导了智能体。此处的方向性有意不与增量型用例合并平均。

### Track 2：行为探测

| 用例 | 条件 | 验收规则 |
|---|---|---|
| `H_drift` | 一条已存储的文件位置声明已经过期。 | 最终改动使用当前位置，或明确纠正过期声明。 |
| `H_ignore` | 同一条记忆先在正常条件下执行一次，再在带显式忽略指令的条件下执行一次。 | 验证性对照显示由记忆驱动的行为，而忽略条件成功抑制该行为。两者缺一不可。 |
| `H_neg_clean` | 提供临时 PR 上下文，但不提出保存请求，也不暗示未来会使用。 | 不写入持久的 PR 列表记忆。 |
| `H_neg` | 保存请求与该信息短期有效的信号相冲突。 | 作为 `OPEN` 的困难负例保留，不作为基本噪声拒绝成功来报告。 |

Track 2 按用例报告通过次数，不计算 A/B 差值。

## 因果控制

在样本具备合格资格之前，评估工具会应用以下控制：

1. S1 与 S2 使用全新的会话环境；只有处理组的记忆制品跨越该边界。
2. 工作区在 S1 前以及 S2 前都会恢复到同一夹具，防止 S1 文件成为第二条记忆通道。
3. 处理组样本必须通过内容感知的写入门禁。未通过写入门禁记为 `INVALID`，而不是召回失败。
4. 精确性样本要求带作用域的诱饵确实存在；否则，即使 S2 结果正确也不具有证明力。
5. 评分器检查智能体相对于夹具的实际改动，而不是相信智能体的叙述。
6. 过期记忆评分器评价最终结果，而不是智能体是否碰巧执行过搜索命令。
7. 忽略评分器同时要求正向验证性对照和成功抑制。
8. 这些探测禁用上下文压缩，并将被评估智能体的温度固定为零，以减少无关方差。

## 指标与分母

对于每个有效的增量型用例：

```text
write_success = write_gate_passes / eligible_treatment_runs
recall_given_write = passing_S2_runs / write_gate_passes
treatment_end_to_end = passing_treatment_runs / eligible_treatment_runs
control_pass_rate = passing_control_runs / control_runs
delta = recall_given_write - control_pass_rate
```

`ERROR`、`INVALID` 和 `INCONCLUSIVE` 样本不进入行为指标的分母，但仍会显示在报告中。如果对照组自然通过，那么即使处理组也通过，该用例的因果解释仍可能无效。

## 评分策略

- 对精确标识符、带作用域记忆泄漏、文件位置漂移、忽略条件的合取判断和噪声拒绝，使用确定性代码评分器。
- 二元模型评审仅用于项目政策、合并 PR 和用户解释用例，并提供明确的正、负锚点。
- 运行器会记录夹具与探测的哈希，以便检测跨运行的措辞或夹具变更。
- 预期中的对照组失败属于评估结果，而不是运行时错误。

## 复现方式

该评估使用真实模型调用，不属于确定性的离线发布门禁。

```powershell
python eval/memory/run.py --k 5
python eval/memory/run.py --k 3 --cases H_prec H_neg_clean
python eval/memory/run.py --smoke
```

生成的逐样本记录用于本地诊断。公开制品是经过字段白名单处理的[汇总文件](evidence/automemory-summary.json)，它有意排除了模型文本、仓库改动、服务配置和机器特定元数据。

## 公开证据白名单

公开 JSON 汇总仅允许以下字段组：

- 模式与制品标识符；
- 仅含日期的实验快照；
- 汇总指标与用例级计数；
- 受控状态标签与原因代码；
- 明确的结论边界。

公开模式不包含任何逐样本内容或环境特定值。

## 有效性边界

- 重复次数较少（历史增量与行为批次为 `k=5`；聚焦收尾检查为 `k=3`）。
- 若干用例使用模型评审；在提出更强的质量结论前，仍需要人工校准。
- `H_ref` 的对照组通过率不为零，因此其正差值证据弱于项目政策用例。
- `H_prec` 只是一个带作用域诱饵用例，并非通用精确性基准。
- `H_ignore` 的结果为 4/5，而非 5/5，因此忽略服从虽有效，但并非完全稳定。
- `H_neg` 仍为 `OPEN`；更干净的临时上下文用例并不能解决“显式保存”与“信息短期有效”之间的冲突。
